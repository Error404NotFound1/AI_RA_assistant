## 1. 角色概述

**系统架构师**拥有需求工程师的所有权限，并额外拥有架构设计相关的全部权限：

| 功能模块 | 权限 |
|---------|------|
| 项目管理 | 创建、查看、编辑（同 RE） |
| 需求管理 | 全部操作（同 RE） |
| **架构推荐** | 发起 AI 架构推荐 |
| **架构方案** | 创建、编辑、状态管理 |
| **组件管理** | 创建、编辑、删除架构组件 |
| **架构评审** | 提交人工评审、发起 AI 评审、更新评审状态 |
| **ADR 管理** | 创建、编辑、删除架构决策记录 |
| **追溯管理** | 发起 AI 追溯映射、手动管理追溯链接 |
| 文档管理 | 生成架构设计文档、PlantUML 图 |
| AI 助手 | 使用 AI 对话助手 |

## 2. 登录与基础操作

### 注册/登录

参考需求工程师手册第 2 节。注册时角色选择"系统架构师"。

### 工作流程概览

系统架构师的典型工作流程：

```
查看已确认需求
    ↓
AI 架构推荐（ADD 方法）
    ↓
调整/管理架构组件
    ↓
提交架构评审 / 发起 AI 评审
    ↓
创建 ADR 记录关键决策
    ↓
AI 追溯映射（需求 → 组件）
    ↓
生成架构文档 & PlantUML 图
```

## 3. AI 辅助架构推荐

### 3.1 前提条件

- 项目中必须有**已确认状态**的需求（由需求工程师确认）
- 需要系统架构师权限

### 3.2 发起架构推荐

1. 在左侧导航栏点击"架构设计"
2. 在顶部选择目标项目
3. 点击"AI 架构推荐"按钮
4. 在弹出的质量属性选择框中，勾选本项目最关注的质量属性：

   | 质量属性 | 适用场景 |
   |---------|---------|
   | 性能 | 高并发、大数据量处理 |
   | 可用性 | 7×24 小时不间断服务 |
   | 安全性 | 涉及敏感数据、金融系统 |
   | 可修改性 | 需求频繁变更、快速迭代 |
   | 可测试性 | 强调质量保证 |
   | 易用性 | 面向普通用户的系统 |
   | 互操作性 | 需要与多个外部系统集成 |
   | 可靠性 | 数据不丢失、高容错 |

5. 点击"开始推荐"，等待 AI 分析（约 10~30 秒）

### 3.3 AI 推荐结果说明

AI 基于 **ADD（属性驱动设计）方法**分析需求并生成推荐，结果包含：

#### 推荐架构模式
每个候选模式包含：
- **适配度评分**（0-100）：与项目需求的匹配程度
- **优势**：该模式的核心优点
- **劣势**：需要注意的局限性
- **推荐理由**：基于项目需求的具体分析

常见架构模式说明：

| 模式 | 适用场景 | 复杂度 |
|------|---------|--------|
| 分层架构 | 中小型企业应用 | 低 |
| 微服务架构 | 大规模、高并发系统 | 高 |
| 事件驱动架构 | 异步处理、解耦系统 | 中 |
| MVC 架构 | Web 应用、CRUD 系统 | 低 |
| 六边形架构 | 强调可测试性、可替换性 | 中 |

#### 组件划分
AI 自动生成初始组件列表，包含每个组件的名称、类型、职责、接口和依赖关系。

#### 技术栈推荐
AI 基于架构模式和需求特点推荐具体技术选型。

#### 质量属性验证
确认推荐方案如何满足各质量属性要求。

## 4. 架构方案管理

### 4.1 查看架构方案列表

AI 推荐后，方案会出现在架构方案列表中。每个方案显示：
- 方案名称和架构模式
- 版本号（手动编辑后自动递增）
- 状态标签

### 4.2 方案状态流转

点击方案卡片右上角的 ✓ 按钮推进状态：

```
待评审（proposed）
    ↓ 点击 ✓
已选定（selected）
    ↓ 点击 ✓
已评审（reviewed）
    ↓ 点击 ✓
已确认（confirmed）
```

> 建议：完成组件管理、评审和 ADR 后，再将方案状态推进至"已确认"。

### 4.3 展开方案详情

点击方案卡片右上角的展开按钮（▼），进入四个功能标签页：

- **概览**：质量评分、推荐模式详情、技术栈、生成操作
- **组件管理**：管理可编辑的架构组件
- **评审**：查看和提交架构评审记录
- **ADR**：管理架构决策记录

## 5. 架构组件管理

组件管理是对 AI 自动生成组件的人工调整和完善。

### 5.1 查看组件列表

展开方案详情 → 切换到"组件管理"标签页，显示所有已创建的可编辑组件。

> **注意**：概览 Tab 中"AI 推荐组件"是只读的，真正可编辑管理的组件在"组件管理"Tab 中。

### 5.2 添加新组件

1. 在"组件管理"标签页点击"添加组件"按钮
2. 填写组件信息：
   - **组件名称**（必填）：如"用户认证服务"、"API 网关"
   - **组件类型**：选择适合的类型（service/gateway/database/cache/frontend/backend/middleware/queue/other）
   - **职责描述**：详细描述该组件负责什么功能
3. 点击"添加"保存

### 5.3 编辑组件

点击组件右侧的编辑按钮（✏），修改名称、类型或职责，点击"保存"。

### 5.4 删除组件

点击组件右侧的删除按钮（🗑），在确认对话框中点击"删除"。

> **注意**：删除组件会同时删除该组件所有的追溯链接。

## 6. 架构评审（人工 & AI）

### 6.1 人工提交评审

1. 展开方案详情 → 切换到"评审"标签页
2. 点击"提交评审"按钮
3. 选择评分（1-5 星）
4. 填写评审意见（支持写优点、问题、改进建议）
5. 点击"提交评审"

### 6.2 AI 架构评审

AI 将对架构方案进行全面的自动化评审，生成专业的评审报告：

1. 在"评审"标签页点击"AI评审"按钮（或在"概览"标签页点击"AI 架构评审"）
2. 等待 AI 分析（约 10~30 秒）
3. 评审报告弹窗显示：

**报告内容包含：**

| 报告节 | 内容 |
|--------|------|
| 综合评分 | 0-10 分整体评价 |
| 质量属性评估 | 各质量属性满足情况 |
| 架构模式适配性 | 所选模式是否适合当前需求 |
| 发现缺陷 | 架构设计中的问题和风险 |
| 改进建议 | 具体的优化方向和建议 |

AI 评审结果会自动保存为一条评审记录（标记为"[AI评审]"前缀）。

### 6.3 管理评审记录

每条评审记录可以：

- **推进状态**：点击刷新按钮（↻）推进 `待处理 → 已响应 → 已解决`
- **删除**：点击删除按钮（🗑）

> 建议：对于 AI 评审发现的缺陷，在修改方案后将评审状态推进至"已解决"。

## 7. 架构决策记录（ADR）

ADR 用于记录重要的架构决策及其背景，是架构文档的重要组成部分。

### 7.1 什么时候创建 ADR？

以下情况应创建 ADR：
- 选择某个架构模式而非其他模式时（为什么选分层架构而不是微服务）
- 选定某个关键技术时（为什么选 PostgreSQL 而非 MongoDB）
- 对架构的重大修改时
- 放弃某个设计决策时

### 7.2 创建 ADR

1. 展开方案详情 → 切换到"ADR"标签页
2. 点击"创建 ADR"按钮
3. 填写 ADR 信息：

   - **标题**（必填）：简洁描述决策，如"选择 PostgreSQL 作为主数据库"
   - **背景（Context）**（必填）：描述做出该决策的背景、问题和约束条件
   - **决策（Decision）**（必填）：明确说明做出了什么决策
   - **后果（Consequences）**（可选）：描述该决策带来的正面和负面影响

4. 点击"创建 ADR"

### 7.3 管理 ADR

**编辑 ADR：**
点击 ADR 右侧的编辑按钮（✏），可修改所有字段，并更新 ADR 状态：

| 状态 | 含义 |
|------|------|
| 草案（proposed） | 决策尚在讨论中 |
| 已接受（accepted） | 决策已被团队采纳 |
| 已废弃（deprecated） | 决策已过时，不再适用 |
| 已替代（superseded） | 被新的决策所取代 |

**删除 ADR：**
点击删除按钮（🗑），在确认对话框中点击"删除"。

## 8. AI 追溯映射

追溯映射建立需求与架构组件之间的关联关系，是架构设计完整性的重要保证。

### 8.1 AI 自动追溯映射

1. 在架构设计页面顶部点击"AI 追溯映射"按钮
2. AI 自动分析已确认需求与已有组件，建立映射关系
3. 映射结果可在"可追溯性矩阵"页面查看

### 8.2 手动管理追溯链接

如需手动调整 AI 生成的追溯链接（尚需通过"可追溯性矩阵"页面操作）：
- 查看每条需求映射到哪些组件
- 确认映射关系的置信度和映射类型

### 8.3 映射类型说明

| 映射类型 | 含义 |
|---------|------|
| direct | 需求直接由该组件实现 |
| indirect | 组件间接支持该需求 |
| constraint | 需求对组件施加了约束 |

## 9. 可追溯性矩阵

### 9.1 访问追溯矩阵

在左侧导航栏点击"追溯矩阵"。

### 9.2 查看矩阵内容

选择项目后，矩阵展示：
- **需求总数**、**已覆盖需求数**、**覆盖率百分比**
- 每条需求对应的用户故事、用例、关联的架构组件
- 覆盖状态（已覆盖 / 未覆盖）

### 9.3 提升覆盖率

对于"未覆盖"的需求：
1. 检查是否已进行 AI 追溯映射
2. 必要时手动添加组件或重新进行 AI 追溯映射
3. 确保所有已确认需求都有对应的架构组件覆盖

## 10. 架构文档与 PlantUML 图生成

### 10.1 生成架构设计文档

1. 展开方案详情 → 切换到"概览"标签页
2. 点击"生成架构文档"按钮
3. AI 自动生成符合 IEEE 1471 标准的 Markdown 架构设计文档
4. 文档在页面内展示，可直接阅读

### 10.2 生成 PlantUML 组件图

1. 展开方案详情 → 切换到"概览"标签页
2. 点击"生成 PlantUML"按钮
3. AI 生成 PlantUML 代码，系统自动渲染为架构图（调用 PlantUML 公共服务）
4. 可点击"查看 PlantUML 源码"展开代码，复制后可在本地 PlantUML 工具中使用

> **注意**：PlantUML 图渲染需要访问 www.plantuml.com，请确保网络可用。如渲染失败，仍可复制源码本地渲染。

### 10.3 将架构文档保存到文档管理

AI 生成的架构文档目前仅在页面内展示，建议手动复制到文档管理模块保存：
1. 复制生成的 Markdown 文本
2. 前往"文档管理"，选择项目
3. 创建新文档（"AI 架构设计文档"），粘贴内容保存

系统架构师拥有需求工程师的所有权限，具体操作请参考：
- **《用户手册-需求工程师版》** 中的第 4~10 节

## 12. 常见问题

**Q：AI 架构推荐提示"没有已确认的需求"？**
> 需求工程师必须先完成需求分析并将需求状态改为"已确认"，架构推荐才能发起。

**Q：方案状态已推进至"已确认"，还能修改吗？**
> 可以，但建议创建新版本而非直接修改已确认方案，以保留历史记录。

**Q：PlantUML 图显示"图片加载失败"？**
> 可能是网络问题，尝试刷新页面。或复制 PlantUML 源码，使用本地工具（如 VSCode PlantUML 插件）渲染。

**Q：AI 追溯映射后，需求覆盖率仍然很低？**
> 检查：① 是否已添加足够的架构组件；② AI 可能未能识别某些需求与组件的关联，可手动在追溯矩阵中调整。

**Q：ADR 应该创建多少条合适？**
> 通常每个重大架构决策一条，一个中等规模项目建议 5~15 条 ADR。质量比数量更重要。

**Q：如何与需求工程师协作？**
> 系统通过状态流转实现协作：需求工程师确认需求 → 架构师基于确认需求设计架构 → 架构师反馈需求问题 → 需求工程师修改 → 循环迭代。
