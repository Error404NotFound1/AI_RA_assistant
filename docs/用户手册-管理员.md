## 1. 角色概述

**系统管理员**是系统最高权限角色，拥有所有功能的访问权限。

| 功能模块 | 权限 |
|---------|------|
| 用户管理 | 查看、创建、角色分配、启用/禁用所有用户 |
| 系统仪表盘 | 查看系统整体运行统计 |
| AI 使用统计 | 查看 AI 调用频次、需求覆盖率等指标 |
| 操作日志 | 查询、筛选、导出所有操作日志 |
| 项目管理 | 管理所有用户的所有项目 |
| 需求管理 | 管理所有项目的需求 |
| 架构设计 | 管理所有项目的架构方案 |
| 文档管理 | 管理所有文档 |

## 2. 管理员登录

### 默认管理员账号

系统初始化后自动创建管理员账号：

| 字段 | 值 |
|------|-----|
| 用户名 | `admin` |
| 密码 | `admin123456` |

> **⚠️ 安全提示**：首次登录后请立即修改默认密码。

### 登录步骤

1. 访问 http://localhost:3000/login
2. 输入用户名 `admin`，密码 `admin123456`
3. 点击"登录"按钮
4. 成功后跳转至工作台仪表盘
5. 管理员专属的"管理后台"入口在左侧导航栏底部

## 3. 用户管理

### 3.1 访问用户管理

1. 在左侧导航栏点击"管理后台"（仅管理员可见）
2. 进入管理面板后，默认显示用户列表

### 3.2 查看用户列表

用户列表显示所有已注册用户的信息：

| 字段 | 说明 |
|------|------|
| 用户名 | 用户的登录名 |
| 邮箱 | 注册邮箱 |
| 姓名 | 用户显示名 |
| 角色 | requirement_engineer / system_architect / admin |
| 状态 | 活跃 / 已禁用 |
| 注册时间 | 账号创建时间 |

### 3.3 修改用户角色

1. 在用户列表中找到目标用户
2. 点击该用户所在行的"修改角色"按钮（或下拉菜单）
3. 选择新角色：
   - **需求工程师（RE）**：可进行需求分析相关操作
   - **系统架构师（SA）**：可进行架构设计相关操作（含 RE 所有权限）
   - **管理员（Admin）**：拥有所有权限
4. 确认修改

> **注意**：角色修改即时生效，用户下次发起请求时自动应用新权限（无需重新登录）。

### 3.4 启用/禁用用户

1. 在用户列表中找到目标用户
2. 点击"启用"或"禁用"按钮
3. 确认操作

**禁用用户的效果**：
- 被禁用的用户无法登录系统
- 已登录的用户 Token 在过期后无法刷新（默认 30 分钟后自动下线）
- 禁用用户不会删除其数据

**恢复被禁用用户**：再次点击"启用"按钮即可恢复访问。

### 3.5 用户安全注意事项

- 不要随意提升用户为管理员角色
- 离职员工应及时禁用账号
- 如发现异常登录，可立即禁用相关账号

## 5. AI 使用统计

### 5.1 访问 AI 统计

在管理后台点击"AI 统计"标签页。

### 5.2 统计指标说明

| 指标 | 计算说明 |
|------|---------|
| AI 分析总次数 | 所有需求的 AI 分析调用总次数 |
| 架构推荐总次数 | AI 架构推荐调用总次数 |
| 文档生成总次数 | AI 文档生成调用总次数 |
| 需求覆盖率 | 已分析需求数 / 总需求数 × 100% |
| 平均每项目需求数 | 总需求数 / 项目数 |
| 已确认需求占比 | 已确认需求数 / 总需求数 × 100% |

### 5.3 如何使用统计数据

- **AI 调用频次低**：可能是用户不了解 AI 功能，需要培训
- **需求覆盖率低**：可能有项目需求未经 AI 分析，质量存在风险
- **已确认需求占比低**：可能是需求工程师未及时审核确认 AI 分析结果

## 6. 操作日志

系统自动记录关键操作日志，帮助管理员追溯系统使用情况。

### 6.1 访问操作日志

在管理后台点击"操作日志"标签页。

### 6.2 日志查询与筛选

支持按以下条件筛选日志：

| 筛选条件 | 说明 |
|---------|------|
| 操作类型（action） | 如 analyze、recommend、generate_doc 等 |
| 用户 ID | 查看某个用户的所有操作 |
| 目标类型（target_type） | 如 requirement、architecture、document |
| 开始时间 | 日志时间范围起点 |
| 结束时间 | 日志时间范围终点 |
| 每页条数 | 10 / 20 / 50 条 |

### 6.3 记录的操作类型

| 操作类型 | 触发场景 |
|---------|---------|
| analyze | AI 需求分析 |
| recommend | AI 架构推荐 |
| auto_map | AI 追溯映射 |
| generate_doc | AI 生成需求规格文档 |
| generate_arch_doc | AI 生成架构设计文档 |
| generate_plantuml | AI 生成 PlantUML 图 |
| ai_review | AI 架构评审 |
| update_review | 更新评审状态 |
| update_adr | 更新 ADR |
| create_component | 创建架构组件 |
| view_requirement | 查看需求详情（统计类） |
| view_architecture | 查看架构方案详情（统计类） |

> **记录范围**：写操作（创建/修改/删除）全部记录；读操作中，仅统计类访问（如查看分析结果、查看架构详情）会被记录。

### 6.4 导出日志

1. 设置筛选条件
2. 点击"导出日志"按钮
3. 系统自动下载 CSV 格式日志文件

## 8. 系统安全管理

### 8.1 API 密钥管理

AI API 密钥存储在服务器 `backend/.env` 文件中，**不会**通过任何 API 接口暴露给前端或用户。

如需更换 API 密钥：
1. 编辑服务器上的 `backend/.env` 文件
2. 修改 `DEEPSEEK_API_KEY` 或 `OPENAI_API_KEY`
3. 重启后端服务：`docker compose restart backend`

### 8.2 JWT 密钥管理

JWT 密钥（`SECRET_KEY`）用于签发用户 Token，如需轮换：
1. 生成新密钥：`python -c "import secrets; print(secrets.token_hex(32))"`
2. 更新 `backend/.env` 中的 `SECRET_KEY`
3. 重启后端（所有已登录用户将被强制退出）

### 8.3 数据备份建议

定期备份 PostgreSQL 数据：

```bash
# Docker Compose 环境备份
docker compose exec postgres pg_dump -U postgres ai_se_assistant > backup_$(date +%Y%m%d).sql
```

### 8.4 安全检查清单

管理员应定期检查：

- [ ] 是否有长期未登录的用户账号（建议禁用）
- [ ] 是否有异常的大量 AI 调用（查看操作日志）
- [ ] API 密钥是否定期轮换
- [ ] 默认管理员密码是否已修改
- [ ] 数据库是否已备份

## 9. 常见问题

**Q：如何创建新的普通用户？**
> 目前系统通过用户自助注册创建账号，管理员可在注册后修改其角色。如需禁止公开注册，需在后端代码中配置（当前版本不支持界面配置此项）。

**Q：如何批量导入用户？**
> 当前版本不支持批量导入，需要用户逐个自助注册，管理员随后调整角色。

**Q：日志保留多久？**
> 系统目前不自动清理日志，所有操作日志永久保存在数据库中。如数据库容量紧张，可手动执行 SQL 清理旧日志。

**Q：如何重置某用户的密码？**
> 当前版本不支持管理员重置其他用户密码，用户需通过"忘记密码"功能（如系统配置了邮件服务）或联系管理员直接操作数据库重置。

**Q：系统出现异常，如何排查？**
> 1. 查看后端日志：`docker compose logs backend --tail 100`
> 2. 查看操作日志中的异常操作
> 3. 检查 API 密钥是否有效
> 4. 检查数据库连接是否正常

**Q：如何查看某个用户做了哪些操作？**
> 在操作日志页面，按"用户 ID"筛选，先在用户列表中获取该用户的 UUID，然后填入筛选条件。

**Q：管理员也需要 AI 功能吗？**
> 管理员主要负责系统管理，AI 功能由需求工程师和架构师使用。但管理员可以代替任何角色操作，用于调试或演示。
