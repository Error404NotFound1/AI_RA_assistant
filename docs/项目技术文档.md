## 2. 工作流总览

系统包含 **8 个阶段**，其中 **4 处调用 AI**：

| 阶段 | 是否调用 AI | AI 调用次数 | 说明 |
|------|:-----------:|:-----------:|------|
| Phase 1: 用户认证 | 否 | 0 | JWT 令牌 + RBAC 角色校验 |
| Phase 2: 项目管理 | 否 | 0 | CRUD + 成员管理 |
| Phase 3: 需求分析 | **是** | **3 次** | 提取 → 分类 → 质量评估（链式调用） |
| Phase 4: 需求确认 | 否 | 0 | 人工审核，状态流转 |
| Phase 5: 架构推荐 | **是** | **1 次** | ADD 方法架构推荐 |
| Phase 6: 架构评审 | 否 | 0 | 人工评审 + ADR 记录 |
| Phase 7: 追溯映射 | **是** | **1 次** | 需求-组件自动映射 |
| Phase 8: 管理员 | 否 | 0 | 用户管理 + 统计 |

**AI 调用总计：每轮完整工作流 5 次 LLM 调用**

### 1.2 RBAC 权限模型

权限检查通过 FastAPI 依赖注入实现，代码位于 `backend/app/core/deps.py`：

```python
# 角色枚举定义 (app/models/user.py)
class UserRole(str, Enum):
    REQUIREMENT_ENGINEER = "RE"   # 需求工程师
    SYSTEM_ARCHITECT = "SA"        # 系统架构师
    ADMIN = "admin"                # 管理员

# 权限检查依赖工厂
def require_role(allowed_roles: list[UserRole]):
    async def check_role(current_user: CurrentUser) -> User:
        if current_user.role not in allowed_roles:
            raise HTTPException(status_code=403, detail="权限不足")
        return current_user
    return check_role

# 预定义权限级别
RequireRE    = RE + SA + Admin    # 可发起 AI 需求分析
RequireSA    = SA + Admin         # 可推荐架构、提交评审、创建 ADR
RequireAdmin = Admin              # 系统管理
```

### 1.3 JWT 令牌结构

```json
// Access Token Payload
{
  "exp": "2026-05-29T13:00:00Z",   // 过期时间 (30分钟)
  "sub": "uuid-of-user",            // 用户 ID
  "type": "access"                  // 令牌类型
}

// Refresh Token Payload
{
  "exp": "2026-06-05T12:30:00Z",   // 过期时间 (7天)
  "sub": "uuid-of-user",
  "type": "refresh"
}
```

### 1.4 前端认证流程

位于 `frontend/src/lib/auth-store.ts`：

```
1. login(username, password)
   → 调用 POST /auth/login
   → 存储 access_token / refresh_token 到 localStorage
   → 调用 GET /auth/me 获取用户信息
   → 设置 Zustand store: { user, isAuthenticated: true }

2. 页面加载时 initialize()
   → 检查 localStorage 是否有 token
   → 有则调用 GET /auth/me 恢复用户状态
   → 无则重定向到 /login

3. Axios 拦截器 (frontend/src/lib/api.ts)
   → 请求拦截: 自动添加 Authorization header
   → 响应拦截: 401 时清除 token 并跳转登录页
```

### 3.3 Step 1: 需求提取（AI 调用 #1）

**目的**: 从自然语言需求描述中提取结构化的功能需求 (FR) 和非功能需求 (NFR)。

**代码位置**: `requirements.py` 第 116-127 行

```python
# 构建提示词
extractor_prompt = EXTRACTOR_USER_TEMPLATE.format(
    project_name=project.name,
    project_description=project.description or "无",
    requirement_text=req_text,
)

# 调用 LLM
extract_result = await provider.complete(
    EXTRACTOR_SYSTEM_PROMPT,    # 系统提示词
    extractor_prompt,           # 用户提示词
    # temperature 默认 0.3 (低温度=更确定/一致的输出)
)
```

**使用的提示词**:

- System Prompt: `EXTRACTOR_SYSTEM_PROMPT`（见 [提示词参考](#1-需求提取-extractor)）
- User Template: `EXTRACTOR_USER_TEMPLATE`，填充变量: `{project_name}`, `{project_description}`, `{requirement_text}`

**LLM 调用参数**:

| 参数 | 值 | 说明 |
|------|-----|------|
| temperature | 0.3 | 低温度，确保输出稳定一致 |
| response_format | `{"type": "json_object"}` | 强制 JSON 模式输出 |
| messages | system + user | 标准双角色对话 |

**期望输出格式**:

```json
{
  "functional_requirements": [
    {
      "id": "FR-001",
      "title": "需求标题",
      "description": "需求详细描述",
      "type": "业务/技术/约束",
      "source": "来源说明"
    }
  ],
  "non_functional_requirements": [
    {
      "id": "NFR-001",
      "title": "需求标题",
      "description": "需求详细描述",
      "category": "性能/安全/可用性/可靠性/可扩展性",
      "source": "来源说明"
    }
  ]
}
```

**提取规则**（嵌入在 System Prompt 中）:
1. 功能需求描述系统"做什么"，非功能需求描述系统"做到什么程度"
2. 识别隐含的非功能需求（性能/安全/可用/可靠/可扩展关键词）
3. 每条需求只包含一个功能点（单一性原则）
4. 标注需求来源

### 3.4 Step 2: MoSCoW 优先级分类（AI 调用 #2）

**目的**: 使用 MoSCoW 方法对提取的需求进行优先级排序。

**代码位置**: `requirements.py` 第 129-137 行

```python
# 将 Step 1 的结果序列化为 JSON 字符串，作为 Step 2 的输入

# 将分类结果合并到总分析结果中
```

**使用的提示词**:

- System Prompt: `CLASSIFIER_SYSTEM_PROMPT`
- User Template: `CLASSIFIER_USER_TEMPLATE`，填充变量: `{project_name}`, `{requirements_json}`

**MoSCoW 方法定义**（嵌入在 System Prompt 中）:
- **Must**: 必须有，系统不可交付的核心需求
- **Should**: 应该有，重要但可暂时延后的需求
- **Could**: 可以有，锦上添花的需求
- **Wont**: 暂不考虑，明确排除的需求

**期望输出格式**:

```json

      "id": "FR-001",
```

### 3.5 Step 3: INCOSE 质量评估（AI 调用 #3）

**目的**: 基于 INCOSE 7 项质量属性对需求逐条评估质量。

**代码位置**: `requirements.py` 第 139-146 行

```python
# 构建提示词（使用 Step 1 的结果，不是 Step 2）
quality_prompt = QUALITY_CHECKER_USER_TEMPLATE.format(
    project_name=project.name,
    requirements_json=req_json,   # ← Step 1 的输出
)

# 调用 LLM
quality_result = await provider.complete(
    QUALITY_CHECKER_SYSTEM_PROMPT,
    quality_prompt,
)

# 将质量评估结果合并到总分析结果中
if quality_result.parsed_json:
    analysis["quality_evaluation"] = quality_result.parsed_json
```

**使用的提示词**:

- System Prompt: `QUALITY_CHECKER_SYSTEM_PROMPT`
- User Template: `QUALITY_CHECKER_USER_TEMPLATE`，填充变量: `{project_name}`, `{requirements_json}`

**INCOSE 7 项质量属性**（嵌入在 System Prompt 中）:

| 属性 | 评分方式 | 检查内容 |
|------|----------|----------|
| 完整性 | 0-10 分 | 是否缺少前提条件、约束、验收标准 |
| 一致性 | 通过/不通过 | 需求之间是否矛盾 |
| 可验证性 | 0-10 分 | 是否包含可量化的验收标准 |
| 无歧义性 | 0-10 分 | 检测模糊词汇（量词、形容词、逃避条款、开放式条款） |
| 可追溯性 | 通过/不通过 | 是否关联了业务目标或利益相关者 |
| 可行性 | 0-10 分 | 技术和资源上是否可实现 |
| 单一性 | 通过/不通过 | 是否只描述一个功能（检测"和/同时/以及/并且/或者"） |

**模糊词检测列表**（嵌入在 System Prompt 中）:
- 模糊量词: 一些、任何、几个、许多、很多、几乎、大约、接近
- 模糊形容词: 相关的、常规的、通用的、显著的、灵活的、可扩展的、典型的、充分的、适当的、高效的
- 逃避条款: 尽可能、如果可能、如果需要、在适当情况下、在可行的情况下
- 开放式条款: 包括但不限于、等等、诸如此类

## LLM Provider 抽象层详解

### 架构设计

```
              get_llm_provider()
                     │
         ┌───────────┴───────────┐
         │                       │
  LLM_PROVIDER="deepseek"  LLM_PROVIDER="openai"
         │                       │
         ▼                       ▼
  DeepSeekProvider        OpenAIProvider
         │                       │
         └───────────┬───────────┘
                     │
              LLMProvider (ABC)
                     │
         ┌───────────┴───────────┐
         │                       │
    complete()              stream()
    (同步调用)              (流式调用)
         │                       │
         ▼                       ▼
    返回 LLMResponse        yield str (chunk)
    {                       逐块返回内容
      content: str,
      parsed_json: dict,
      usage: {prompt_tokens,
              completion_tokens,
              total_tokens}
    }
```

### DeepSeek Provider 实现

```python
# deepseek_provider.py
class DeepSeekProvider(LLMProvider):
    def __init__(self):
        self.model = settings.DEEPSEEK_MODEL  # "deepseek-chat"
        self.client = AsyncOpenAI(
            api_key=settings.DEEPSEEK_API_KEY,
            base_url=settings.DEEPSEEK_BASE_URL,  # "https://api.deepseek.com"
        )

    async def complete(self, system_prompt, user_prompt, temperature=0.3):
        response = await self.client.chat.completions.create(
            model=self.model,
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_prompt},
            ],
            temperature=temperature,
            response_format={"type": "json_object"},  # ← 强制 JSON 输出
        )
        content = response.choices[0].message.content
        parsed_json = json.loads(content)  # 尝试解析为 JSON
        return LLMResponse(content=content, parsed_json=parsed_json, usage=...)
```

### 关键设计决策

1. **temperature=0.3**: 所有 AI 调用使用低温度，确保输出稳定、一致、可解析
2. **response_format=json_object**: 强制 LLM 输出合法 JSON，便于程序解析
3. **JSON 解析容错**: 即使 LLM 返回非 JSON，`parsed_json` 为 None 时会抛出 HTTP 500 错误
4. **Provider 切换**: 修改环境变量 `LLM_PROVIDER` 即可切换，无需改代码

## 完整提示词工程参考

所有提示词定义在 `backend/app/llm/prompts/requirement_extractor.py` 中。

### 1. 需求提取 (EXTRACTOR)

**变量**: `{project_name}`, `{project_description}`, `{requirement_text}`

**System Prompt 核心内容**:

```
你是一位专业的需求工程师，负责从自然语言描述中提取功能需求和非功能需求。

输出格式：JSON，包含 functional_requirements[] 和 non_functional_requirements[]

提取规则：
1. 功能需求描述系统"做什么"，非功能需求描述系统"做到什么程度"
2. 识别隐含的非功能需求：
   - 性能类关键词：快速、响应、实时、并发、吞吐量
   - 安全类关键词：加密、防护、权限、认证、授权
   - 可用类关键词：简洁、直观、适配、友好、易用
   - 可靠类关键词：稳定、备份、恢复、容错、高可用
   - 可扩展类关键词：扩展、模块化、插件、配置
3. 每条需求应该只包含一个功能点（单一性原则）
4. 为每条需求标注来源
```

### 2. MoSCoW 分类 (CLASSIFIER)

**变量**: `{project_name}`, `{requirements_json}`

**System Prompt 核心内容**:

```
你是一位需求分类专家，使用 MoSCoW 方法对需求进行优先级排序。

MoSCoW 方法：
- Must：必须有，系统不可交付的核心需求
- Should：应该有，重要但可暂时延后的需求
- Could：可以有，锦上添花的需求
- Wont：暂不考虑，明确排除的需求

输出格式：JSON，包含 classified_requirements[{id, priority, rationale}]
```

### 3. INCOSE 质量评估 (QUALITY_CHECKER)

**变量**: `{project_name}`, `{requirements_json}`

**System Prompt 核心内容**:

```
你是一位需求质量评估专家，依据 INCOSE 7 项质量属性评估需求质量。

7 项属性：
1. 完整性 (0-10分) — 检查是否缺少前提条件、约束、验收标准
2. 一致性 (通过/不通过) — 检测需求间逻辑冲突
3. 可验证性 (0-10分) — 检查可量化验收标准
4. 无歧义性 (0-10分) — 检测模糊词汇：
   - 模糊量词：一些、任何、几个、许多...
   - 模糊形容词：相关的、常规的、通用的...
   - 逃避条款：尽可能、如果可能...
   - 开放式条款：包括但不限于、等等...
5. 可追踪性 (通过/不通过) — 是否关联业务目标
6. 可行性 (0-10分) — 技术和资源是否可实现
7. 单一性 (通过/不通过) — 检测复合连接词："和"、"同时"...

输出格式：JSON，包含 evaluations[{requirement_id, 7项评分, issues[], suggestions[]}]
```

### 4. 架构推荐 (ARCH_RECOMMENDER)

**变量**: `{project_name}`, `{project_description}`, `{team_size}`, `{budget}`, `{requirements_json}`, `{quality_attributes}`

**System Prompt 核心内容**:

```
你是一位资深软件架构师，基于属性驱动设计（ADD）方法推荐架构方案。

ADD 方法步骤：
1. 识别优先级质量属性场景
2. 生成候选架构模式
3. 组件划分与职责分配
4. 技术栈推荐
5. 质量属性验证

架构模式知识库：
- 分层架构：中小型项目，结构清晰，维护成本低
- 微服务架构：大型项目，独立部署，技术栈灵活
- 事件驱动架构：实时处理，解耦度高
- MVC架构：Web应用，关注点分离
- 六边形架构：多端适配，核心业务隔离

输出格式：JSON，包含 quality_attributes[], recommended_patterns[],
components[], tech_stack{}, quality_verification[]
```

### 5. 追溯映射 (TRACEABILITY_MAPPER)

**变量**: `{requirements_json}`, `{components_json}`

**System Prompt 核心内容**:

```
你是一位需求追踪专家，负责建立需求与架构组件之间的映射关系。

映射类型：
- direct（直接实现）：组件直接满足需求功能
- indirect（间接支撑）：组件为需求提供基础设施支撑
- constraint（约束影响）：组件约束了需求的实现方式

约束：
1. 每条功能需求至少映射到一个组件
2. confidence < 0.7 的映射需标注"需人工确认"
3. 未覆盖的需求列入 uncovered_requirements

输出格式：JSON，包含 mappings[{requirement_id, component_name,
mapping_type, confidence, rationale}], uncovered_requirements[],
coverage_percentage
```

### 6. 文档生成 (DOC_GENERATOR)

**变量**: `{project_name}`, `{analysis_result_json}`

**System Prompt 核心内容**:

```
你是一位需求规格说明书编写专家，将分析结果组装为完整的 SRS 文档。

输出 Markdown 格式文档，包含章节：
1. 概述（项目背景、目标、范围）
2. 功能需求列表（编号、标题、描述、优先级）
3. 非功能需求列表（编号、标题、描述、类别）
4. 用户故事（As a... I want... So that...）
5. 用例描述（参与者、前置条件、主流程、异常流程、后置条件）
6. 质量评估摘要（7 项属性的评估结果汇总）

约束：
- 使用 IEEE 830 推荐的章节结构
- 每条需求必须有唯一编号
- 非功能需求应包含可量化的指标
```

## 完整工作流时间线

以一个典型用户操作为例：

```
时间    操作                      AI调用    系统行为
────    ────                      ────     ──────
T0      用户注册                  -        bcrypt 加密密码，存入 users 表
T1      用户登录                  -        验证密码，签发 JWT 双令牌
T2      创建项目                  -        创建 project + project_member 记录
T3      录入需求 (3条)            -        创建 3 条 requirement 记录 (status=draft)
T4      发起 AI 需求分析          #1       LLM 提取 FR/NFR
                                 #2       LLM MoSCoW 分类
                                 #3       LLM INCOSE 质量评估
                                 -        保存 analysis_result, status→analyzed
T5      人工确认需求              -        status→confirmed
T6      发起 AI 架构推荐          #4       LLM ADD 方法推荐架构
                                 -        保存 ArchitectureSolution + ArchComponent
T7      提交架构评审              -        创建 ArchReview 记录
T8      创建 ADR                 -        创建 ADR 记录
T9      发起 AI 追溯映射          #5       LLM 需求-组件映射
                                 -        保存 TraceabilityLink 记录

总计: 5 次 AI 调用 (LLM API)
```
