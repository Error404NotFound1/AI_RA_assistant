## 2. 工作流总览

系统包含 **10 个阶段**，其中 **6 处调用 AI**（其中 3 处为按需调用）：

| 阶段 | 是否调用 AI | AI 调用次数 | 说明 |
|------|:-----------:|:-----------:|------|
| Phase 1: 用户认证 | 否 | 0 | JWT 令牌 + RBAC 角色校验 |
| Phase 2: 项目管理 | 否 | 0 | CRUD + 成员管理 |
| Phase 3: 需求分析 | **是** | **4 次** | 提取(1) + 并行：分类/质量评估/用例生成(3) |
| Phase 4: 需求确认 | 否 | 0 | 人工审核，状态流转 |
| Phase 5: 架构推荐 | **是** | **1 次** | ADD 方法架构推荐 |
| Phase 6: 架构评审 | 否 | 0 | 人工评审 + ADR 记录 |
| Phase 7: 追溯映射 | **是** | **1 次** | 需求-组件自动映射 |
| Phase 8: 文档生成 | **是（按需）** | **1 次** | AI 生成 SRS 文档（Markdown 输出） |
| Phase 9: 架构文档 | **是（按需）** | **2 次** | 架构设计文档 + PlantUML 组件图 |
| Phase 10: 管理员 | 否 | 0 | 用户管理 + 统计 |

**AI 调用总计**：每轮完整工作流 6 次 LLM 调用（含文档/架构文档生成可达 9 次）

> **并行优化**：Phase 3 中 Step 2/3/4 通过 `asyncio.gather` 并行执行，实际延迟约 2×LLM（而非 4×LLM）

### 1.2 RBAC 权限模型

权限检查通过 FastAPI 依赖注入实现，代码位于 `backend/app/core/deps.py`：

```python
# 角色枚举定义 (app/models/user.py)
class UserRole(str, Enum):
    REQUIREMENT_ENGINEER = "RE"   # 需求工程师
    SYSTEM_ARCHITECT = "SA"        # 系统架构师
    ADMIN = "admin"                # 管理员

# 权限检查依赖工厂
def require_role(allowed_roles: list[UserRole]):
    async def check_role(current_user: CurrentUser) -> User:
        if current_user.role not in allowed_roles:
            raise HTTPException(status_code=403, detail="权限不足")
        return current_user
    return check_role

# 预定义权限级别
RequireRE    = RE + SA + Admin    # 可发起 AI 需求分析
RequireSA    = SA + Admin         # 可推荐架构、提交评审、创建 ADR
RequireAdmin = Admin              # 系统管理
```

### 1.3 JWT 令牌结构

```json
// Access Token Payload
{
  "exp": "2026-05-29T13:00:00Z",   // 过期时间 (30分钟)
  "sub": "uuid-of-user",            // 用户 ID
  "type": "access"                  // 令牌类型
}

// Refresh Token Payload
{
  "exp": "2026-06-05T12:30:00Z",   // 过期时间 (7天)
  "sub": "uuid-of-user",
  "type": "refresh"
}
```

### 1.4 前端认证流程

位于 `frontend/src/lib/auth-store.ts`：

```
1. login(username, password)
   → 调用 POST /auth/login
   → 存储 access_token / refresh_token 到 localStorage
   → 调用 GET /auth/me 获取用户信息
   → 设置 Zustand store: { user, isAuthenticated: true }

2. 页面加载时 initialize()
   → 检查 localStorage 是否有 token
   → 有则调用 GET /auth/me 恢复用户状态
   → 无则重定向到 /login

3. Axios 拦截器 (frontend/src/lib/api.ts)
   → 请求拦截: 自动添加 Authorization header
   → 响应拦截: 401 时清除 token 并跳转登录页
```

### 3.3 Step 1: 需求提取（AI 调用 #1）

**目的**: 通过三阶段工作流从自然语言需求描述中提取结构化的功能需求 (FR)、非功能需求 (NFR) 和用户故事。

**代码位置**: `requirements.py` 第 120-131 行

```python
# 构建提示词
extractor_prompt = EXTRACTOR_USER_TEMPLATE.format(
    project_name=project.name,
    project_description=project.description or "无",
    requirement_text=req_text,
)

# 调用 LLM
extract_result = await provider.complete(
    EXTRACTOR_SYSTEM_PROMPT,    # 系统提示词
    extractor_prompt,           # 用户提示词
    # temperature 默认 0.3 (低温度=更确定/一致的输出)
)

analysis = extract_result.parsed_json
```

**使用的提示词**:

- System Prompt: `EXTRACTOR_SYSTEM_PROMPT`（见 [提示词参考](#1-需求提取-extractor)）
- User Template: `EXTRACTOR_USER_TEMPLATE`，填充变量: `{project_name}`, `{project_description}`, `{requirement_text}`

**EXTRACTOR 三阶段工作流**（嵌入在 System Prompt 中）:

| 阶段 | 说明 |
|------|------|
| 阶段一：意图理解 | 分析用户的业务意图和目标，推断核心业务目标、业务领域、主要利益相关者、关键假设 |
| 阶段二：需求补全 | 基于意图理解，补全用户未明确提及但隐含需要的需求（如登录→密码重置/会话管理） |
| 阶段三：结构化提取 | 将理解和补全后的需求提取为结构化的 FR/NFR + 用户故事 |

**LLM 调用参数**:

| 参数 | 值 | 说明 |
|------|-----|------|
| temperature | 0.3 | 低温度，确保输出稳定一致 |
| json_mode | True | 强制 JSON 模式输出 |
| messages | system + user | 标准双角色对话 |

**期望输出格式**:

```json
{
  "intent_analysis": {
    "business_goal": "用户的核心业务目标",
    "domain": "业务领域",
    "stakeholders": ["利益相关者列表"],
    "assumptions": ["基于意图推断的关键假设"]
  },
  "functional_requirements": [
    {
      "id": "FR-001",
      "title": "需求标题",
      "description": "需求详细描述",
      "type": "业务/技术/约束",
      "source": "explicit 或 inferred",
      "rationale": "需求存在理由，尤其对推断补全的需求需说明"
    }
  ],
  "non_functional_requirements": [
    {
      "id": "NFR-001",
      "title": "需求标题",
      "description": "需求详细描述",
      "category": "性能/安全/可用性/可靠性/可扩展性",
      "source": "explicit 或 inferred",
      "rationale": "需求存在理由"
    }
  ],
  "user_stories": [
    {
      "requirement_id": "FR-001",
      "role": "作为...角色",
      "goal": "我希望...功能",
      "benefit": "以便...价值",
      "acceptance_criteria": "可测试的验收标准"
    }
  ]
}
```

**提取规则**（嵌入在 System Prompt 中）:
1. 功能需求描述系统"做什么"，非功能需求描述系统"做到什么程度"
2. 识别隐含的非功能需求（性能/安全/可用/可靠/可扩展关键词）
3. 每条需求只包含一个功能点（单一性原则）
4. 为每条功能需求生成对应的用户故事，包含可测试的验收标准
5. 标注每条需求的来源：`explicit`（用户明确描述）或 `inferred`（基于意图推断补全）
6. 对 `inferred` 需求，在 `rationale` 字段说明推断依据

### 3.4 Step 2/3/4: 并行执行（AI 调用 #2/#3/#4）

Step 1 完成后，Step 2（分类）、Step 3（质量评估）、Step 4（用例生成）**并行执行**，使用 `asyncio.gather`：

**代码位置**: `requirements.py` 第 133-151 行

```python
# 准备并行步骤的 prompt
req_json = json.dumps(analysis, ensure_ascii=False)
classifier_prompt = CLASSIFIER_USER_TEMPLATE.format(
    project_name=project.name, requirements_json=req_json
)
quality_prompt = QUALITY_CHECKER_USER_TEMPLATE.format(
    project_name=project.name, requirements_json=req_json
)
user_stories_json = json.dumps(analysis.get("user_stories", []), ensure_ascii=False)
use_case_prompt = USE_CASE_GENERATOR_USER_TEMPLATE.format(
    project_name=project.name,
    requirements_json=req_json,
    user_stories_json=user_stories_json,
)

# 并行执行 Step 2 (分类), Step 3 (质量评估), Step 4 (用例生成)
results = await asyncio.gather(
    provider.complete(CLASSIFIER_SYSTEM_PROMPT, classifier_prompt),
    provider.complete(QUALITY_CHECKER_SYSTEM_PROMPT, quality_prompt),
    provider.complete(USE_CASE_GENERATOR_SYSTEM_PROMPT, use_case_prompt),
    return_exceptions=True,
)
classify_result, quality_result, use_case_result = results
```

- System Prompt: `CLASSIFIER_SYSTEM_PROMPT`
- User Template: `CLASSIFIER_USER_TEMPLATE`，填充变量: `{project_name}`, `{requirements_json}`

**MoSCoW 方法定义**（嵌入在 System Prompt 中）:
- **Must**: 必须有，系统不可交付的核心需求
- **Should**: 应该有，重要但可暂时延后的需求

#### Step 3: INCOSE 质量评估（AI 调用 #3）

**使用的提示词**: `QUALITY_CHECKER_SYSTEM_PROMPT` + `QUALITY_CHECKER_USER_TEMPLATE`
- 模板变量: `{project_name}`, `{requirements_json}`（Step 1 结果）

**INCOSE 7 项质量属性**（嵌入在 System Prompt 中）:

| 属性 | 评分方式 | 检查内容 |
|------|----------|----------|
| 完整性 | 0-10 分 | 是否缺少前提条件、约束、验收标准 |
| 一致性 | 通过/不通过 | 需求之间是否矛盾 |
| 可验证性 | 0-10 分 | 是否包含可量化的验收标准 |
| 无歧义性 | 0-10 分 | 检测模糊词汇（量词、形容词、逃避条款、开放式条款） |
| 可追溯性 | 通过/不通过 | 是否关联了业务目标或利益相关者 |
| 可行性 | 0-10 分 | 技术和资源上是否可实现 |
| 单一性 | 通过/不通过 | 是否只描述一个功能（检测"和/同时/以及/并且/或者"） |

**模糊词检测列表**（嵌入在 System Prompt 中）:
- 模糊量词: 一些、任何、几个、许多、很多、几乎、大约、接近
- 模糊形容词: 相关的、常规的、通用的、显著的、灵活的、可扩展的、典型的、充分的、适当的、高效的
- 逃避条款: 尽可能、如果可能、如果需要、在适当情况下、在可行的情况下
- 开放式条款: 包括但不限于、等等、诸如此类

## LLM Provider 抽象层详解

### 架构设计

```
              get_llm_provider()  ← 全局单例
                     │
         ┌───────────┴───────────┐
         │                       │
  LLM_PROVIDER="deepseek"  LLM_PROVIDER="openai"
         │                       │
         ▼                       ▼
  DeepSeekProvider        OpenAIProvider
         │                       │
         └───────────┬───────────┘
                     │
              LLMProvider (ABC)
                     │
         ┌───────────┴───────────┐
         │                       │
    complete()              stream()
    (同步调用)              (流式调用)
         │                       │
         ▼                       ▼
    返回 LLMResponse        yield str (chunk)
    {                       逐块返回内容
      content: str,
      parsed_json: dict | None,
      usage: {prompt_tokens,
              completion_tokens,
              total_tokens}
    }
```

### Provider 单例模式

`get_llm_provider()` 返回全局单例实例，避免重复创建 OpenAI 客户端：

```python
# provider.py
_provider_instance: LLMProvider | None = None

def get_llm_provider() -> LLMProvider:
    """根据配置获取 LLM Provider 实例（单例）"""
    global _provider_instance
    if _provider_instance is None:
        if settings.LLM_PROVIDER == "openai":
            from app.llm.openai_provider import OpenAIProvider
            _provider_instance = OpenAIProvider()
        else:
            from app.llm.deepseek_provider import DeepSeekProvider
            _provider_instance = DeepSeekProvider()
    return _provider_instance
```

### complete() 方法与 json_mode

```python
# provider.py 抽象基类
class LLMProvider(ABC):
    @abstractmethod
    async def complete(
        self, system_prompt: str, user_prompt: str,
        temperature: float = 0.3, json_mode: bool = True
    ) -> LLMResponse:
        """同步调用 LLM。json_mode=True 时强制 JSON 输出，False 时允许文本输出"""
        ...

# openai_provider.py 实现
class OpenAIProvider(LLMProvider):
    def __init__(self):
        self.model = settings.OPENAI_MODEL
        client_kwargs = {"api_key": settings.OPENAI_API_KEY, "timeout": 60, "max_retries": 2}
        if settings.OPENAI_BASE_URL:
            client_kwargs["base_url"] = settings.OPENAI_BASE_URL
        self.client = AsyncOpenAI(**client_kwargs)

    async def complete(self, system_prompt, user_prompt, temperature=0.3, json_mode=True):
        kwargs = {
            "model": self.model,
            "messages": [
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_prompt},
            ],
            "temperature": temperature,
        }
        if json_mode:
            kwargs["response_format"] = {"type": "json_object"}  # ← 强制 JSON 输出

        response = await self.client.chat.completions.create(**kwargs)
        content = response.choices[0].message.content or ""

        # 仅在 json_mode=True 时尝试解析 JSON
        parsed_json = None
        if json_mode:
            try:
                parsed_json = json.loads(content)
            except json.JSONDecodeError:
                pass

        return LLMResponse(content=content, parsed_json=parsed_json, usage=...)
```

### 关键设计决策

1. **temperature=0.3**: 所有 AI 调用使用低温度，确保输出稳定、一致、可解析
2. **json_mode 参数**: `json_mode=True` 时强制 JSON 输出（用于需求分析/分类/质量评估）；`json_mode=False` 时允许文本输出（用于文档生成/PlantUML 生成）
3. **JSON 解析容错**: 即使 LLM 返回非 JSON，`parsed_json` 为 None 时会抛出 HTTP 500 错误
4. **Provider 单例**: `get_llm_provider()` 返回全局单例，避免重复创建客户端
5. **超时与重试**: `timeout=60s`, `max_retries=2`
6. **Provider 切换**: 修改环境变量 `LLM_PROVIDER` 即可切换，无需改代码
